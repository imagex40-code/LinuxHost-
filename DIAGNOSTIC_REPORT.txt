PROBLEMS IN LAUNCH FLOW:
  1. Two DIFFERENT proot commands are built for the same operation:
     - Dashboard/Manager "Launch" uses buildLoginCommand() → /bin/bash --login
     - Terminal "Send" uses buildCommand() → /usr/bin/env -i ... /bin/bash -c <command>
     These use different argv, different binds, different behavior.

  2. PROOT_LOADER env var: proot needs this to find the loader. If the loader
     path is wrong or the loader binary is corrupt, proot silently fails with
     a confusing error or segfault.

  3. The loader binary at assets/libexec/proot/loader (18KB) was extracted from
     the Termux proot .deb. It is a separate binary from proot itself — proot
     calls it via execve(). If it's not executable or is corrupt, proot will
     fail with "No such file or directory" or "Exec format error".

  4. --kernel-release string contains literal backslashes. This is intentional
     (proot expects this format), but it's fragile — if any character is wrong,
     programs that call uname() may behave unexpectedly.

  5. --bind=/dev/urandom:/dev/random — if /dev/urandom doesn't exist on the
     host (some Android kernels), this will fail.

  6. Storage binds: /sdcard and /storage/emulated/0 are bind-mounted read-write
     by default. proot-distro uses read-only binds for these. We don't specify
     :ro, which could cause permission issues in some Android versions.

 4. KNOWN ISSUES STILL OPEN
================================================================================

ISSUE A: Proot binary extraction was failing (FIXED BUT NOT VERIFIED ON DEVICE)
  Status:     Fixed in commit 94330de by moving from jniLibs to assets.
  Why unverified: The CI build passed, but the user has not yet tested
                  install on the real device with the new assets path.
  Risk:       If assets/bin/proot is somehow not packaged into the APK
              (extremely unlikely — assets are always included), this will
              still fail. No runtime logging exists to confirm the path.
  What could still be wrong: The proot binary itself might be incompatible
              with the specific Android kernel version on the user's device.
              The binary is from Termux (ARM64, static-ish build), but
              proot uses ptrace() which requires specific kernel config.
              Some OEM kernels disable ptrace for non-root users.

ISSUE B: Navigation was bouncing on error (FIXED BUT NOT VERIFIED)
  Status:     Fixed in commit 94330de. LaunchedEffect now only navigates
              from INSTALLING → success states.
  Why unverified: No device test.
  Risk:       Low — the code logic is straightforward.

ISSUE C: Symlink/hardlink resolution in RootfsExtractor is lossy
  Status:     OPEN, unaddressed.
  Impact:     Some symlinks in the rootfs will point to files that don't exist
              (because the target was a hardlink that wasn't resolved, or the
              target entry appeared later in the tar and was already passed).
              This can break: /bin/sh (often a symlink to dash or bash),
              /usr/bin/env, library symlinks.
  Evidence:   The symlink resolution loop at the END of extract() tries to
              copy symlink targets, but if the target file was never extracted
              (because it was also a symlink, or appeared after the link),
              the symlink becomes a dangling reference.
  Consequence: This is the MOST LIKELY cause of "essential binary missing"
               errors after extraction. The binaries exist in the tarball,
               but their symlink chain is broken.

ISSUE D: Two different proot command builders for the same operation
  Status:     OPEN, by design but problematic.
  Details:    buildLoginCommand() (Dashboard/Manager launch) vs
              buildCommand() (Terminal) produce different argv.
              The Terminal version uses /usr/bin/env -i which strips ALL
              environment variables, then re-adds HOME/USER/TERM.
              The Login version doesn't use env at all — it inherits the
              ProcessBuilder environment (which includes PROOT_TMP_DIR etc.).
  Consequence: Behavior difference between "Launch" and "Terminal" tabs.
               The Login version will have Android env vars leaking in.

ISSUE E: No logging anywhere in the critical path
  Status:     OPEN.
  Details:    ProotInstaller, RootfsExtractor, ProotEngine, TerminalSession
              have ZERO Android Log.d/i/w/e calls. All error messages are
              thrown as RuntimeExceptions and caught by ViewModel, but the
              actual diagnostic info (which file, which path, what the OS
              error was) is lost.
  Consequence: When something fails on the device, the user sees "Installation
               failed" or a generic message. There is no way to diagnose
               without adding log statements.

ISSUE F: ProotEngine.launch() doesn't start a shell session
  Status:     OPEN.
  Details:    ProotEngine.launch() (called from Dashboard "Launch" button)
              starts a proot process with /bin/bash --login but does NOT
              connect any stdin/stdout to it. The process starts and
              immediately exits (no terminal attached, no pipe).
              Only TerminalSession (from Terminal tab) properly manages
              stdin/stdout.
  Consequence: Tapping "Launch" on Dashboard will start proot, set status
               to RUNNING, and then the process will immediately exit.
               The user sees "Running" briefly, then nothing happens.
               They have to go to Terminal tab to actually interact.

ISSUE G: build.gradle.kts has no ndk.abiFilters
  Status:     OPEN but low priority.
  Details:    No abiFilters means the APK includes native libs for ALL ABIs
              (arm64-v8a, armeabi-v7a, x86, x86_64). Since proot is now in
              assets, this doesn't affect the proot binary. But if future
              native libs are added, the APK will be bloated.

ISSUE H: UbuntuBase rootfs URL version mismatch
  Status:     OPEN.
  Details:    ProotEngine rootfsMirrors list references "24.04.4" but the
              Database.kt default version says "26.04". The actual download
              URL is ubuntu-base-24.04.4-base-aarch64.tar.gz.
  Consequence: Minor — DB says 26.04, actual rootfs is 24.04.4. Confusing
               in the UI but doesn't break anything functionally.
